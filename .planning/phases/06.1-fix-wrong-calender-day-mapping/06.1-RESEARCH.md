# Phase 06.1: Fix wrong calendar day mapping - Research

**Researched:** 2026-03-16
**Domain:** Date/time formatting, timezone handling, Luxon library
**Confidence:** HIGH

## Summary

This phase addresses a bug where calendar event responses display the wrong day-of-week name (off by 1 day). For example, Friday events are reported as Saturday. The issue is isolated to the `formatDayName()` function in `src/calendar/timezone.ts`, which uses Luxon's DateTime API to convert YYYY-MM-DD date strings into German weekday names.

The bug is likely caused by timezone handling when parsing date-only strings with `DateTime.fromFormat()`, or a locale/formatting issue with the `cccc` token. Previous weekend query fixes (quick tasks #1, #2) modified LLM prompt logic but didn't touch the display formatting layer, suggesting this is a separate rendering bug.

**Primary recommendation:** Investigate `formatDayName()` function for timezone edge cases when parsing date-only strings, verify Luxon locale configuration, and add comprehensive tests for all 7 weekdays across different query types.

### Anti-Patterns to Avoid

- **Parsing date-only strings without timezone:** Luxon can interpret date-only strings inconsistently. Always pass `{ zone: timezone }` option.
- **Assuming `cccc` format works without setLocale:** Must call `.setLocale("de")` before `.toFormat("cccc")` for German names.
- **UTC vs local timezone confusion:** All display formatting must use `FAMILY_TIMEZONE`, not UTC.

## Common Pitfalls

### Pitfall 1: Date-only string parsing without explicit timezone

**What goes wrong:** When parsing "2026-03-16" without timezone, Luxon may interpret it as midnight in local machine timezone or UTC, leading to off-by-one day errors when formatting.

**Why it happens:** ISO 8601 date-only strings don't specify time or timezone, so Luxon makes assumptions.

**How to avoid:**

- Always pass `{ zone: timezone }` to `DateTime.fromFormat()`
- Current code DOES this (line 94: `{ zone: timezone }`), so likely not root cause

**Warning signs:**

- Events created on one day show as previous/next day
- Bug appears/disappears when testing from different timezones

### Pitfall 2: Locale not applied before formatting

**What goes wrong:** Calling `.toFormat("cccc")` without `.setLocale("de")` returns English weekday names or system locale, not German.

**Why it happens:** Luxon defaults to en-US locale unless explicitly set.

**How to avoid:**

- Always chain `.setLocale("de")` before `.toFormat()`
- Current code DOES this (line 95), so likely not root cause

**Warning signs:**

- Weekday names in wrong language
- Unexpected output like "Monday" instead of "Montag"

### Pitfall 3: Weekday array index mismatch

**What goes wrong:** Luxon's `weekday` property uses ISO 8601 (1=Mon, 7=Sun). If array uses 0=Sun or 0=Mon without empty index 0, names shift.

**Why it happens:** JavaScript Date uses 0=Sunday, Luxon uses 1=Monday. Easy to mix conventions.

**How to avoid:**

- Use Luxon's `.toFormat("cccc")` instead of manual array lookup
- If using array (like in intent.ts), verify index 0 is placeholder

**Warning signs:**

- All weekdays consistently off by 1 day
- Friday shows as Saturday, Saturday as Sunday, etc.

**Investigation priority:** This is the MOST LIKELY root cause. The `formatDayName` function itself looks correct, but verify:

1. Is the date string being passed correctly? (e.g., is "2026-03-19" actually a Friday?)
2. Is Luxon's `toFormat("cccc")` with German locale returning the correct day?
3. Is there a timezone edge case where midnight boundary causes day shift?

### Pitfall 4: Testing without timezone/locale consistency

**What goes wrong:** Tests pass in developer's timezone/locale but fail in production with different settings.

**Why it happens:** Luxon uses system timezone/locale by default if not specified.

**How to avoid:**

- Mock system time with `vi.setSystemTime()` in vitest
- Override Luxon's now with `Settings.now = () => fixedTime`
- Always specify timezone explicitly in tests

**Warning signs:**

- Tests pass locally, fail in CI/CD
- Bug only appears in production environment

### Pitfall 5: Confusion between date parsing formats

**What goes wrong:** Using wrong format string (e.g., "dd-MM-yyyy" instead of "yyyy-MM-dd") causes Luxon to parse incorrectly or return invalid DateTime.

**Why it happens:** Multiple date format standards exist (ISO, European, US).

**How to avoid:**

- Standardize on ISO 8601 (YYYY-MM-DD) for internal date strings
- Verify format string matches actual input format
- Check `dt.isValid` after parsing

**Warning signs:**

- Invalid DateTime errors
- Dates parsing as different values than expected
